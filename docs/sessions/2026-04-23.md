### Evening — TestImages corpus dedup (`TestImages/` untracked, working-tree-only)

Photographer's archive dumped into `TestImages/` as a stress corpus for the ratio analyzer. 5,013 files, 1.6 GB, mostly CMS responsive-image ladders (`{stem}-{width}-{version}.{ext}`). Goal: collapse to one representative per unique image at the best available quality.

- **Pass 1 — filename-based**: regex `^(.+)-(\d+)(?:-(\d+))?\.(jpeg|jpg|webp|png)$`; grouped by `base` (collapsing version suffixes `-1`/`-2`/`-3` as re-exports of one source, per user's domain knowledge); per group kept max-width, jpeg > webp > png tiebreak. **5013 → 645**, freed 1.15 GB.
- **Pass 2 — blanket poster removal**: user flagged all `poster-{hash}-{width}` files as disposable (unrelated graphic-design assets). `find -maxdepth 1 -name 'poster-*' -delete`. **645 → 529**.
- **Pass 3 — perceptual hash**: first tried dHash (64-bit pixel-gradient) → 6 clusters, but Cluster 1 was a 13-file false positive (low-contrast moon/night shots all colliding because dHash's sign-bit signal is noise on flat images). Switched to pHash (32×32 grayscale → 2D DCT → top-left 8×8 sign bits vs median, DC excluded). Pure-Python DCT because system numpy is corrupt (`numpy.arange` missing, PEP-668 blocks pip fix) — 529 files hashed in 3.8s. Hamming ≤8 on 64 bits → **2 clusters surfaced**: `photoindex-1--…` vs `photoindex-…` (real dupe, filename glitch), `gregor-der-zottelkopf-{schiefer-mund|verschaut-sich}` (two intentional expressions of the same subject, not a dupe). User kept both → final **529 files**.

Dry-run lists preserved at `/tmp/aru_dedupe_{delete,keep}.txt` and `/tmp/aru_phash_{delete,clusters}.txt` for audit. No git trail (TestImages is untracked fixture data).

## Learnings / cookbook candidates

- **SUFeedURL bleed-through on xcodeproj clone** — cookbook §34 mentions blanking Sparkle feed URLs but doesn't flag that `SUFeedURL` sits in Info.plist *outside* the pbxproj + Swift source that its sed sweeps touch. Worth adding to §34 gotchas.
- **Sed-by-filename on pbxproj** — works cleanly for bulk file deletion because modern Xcode embeds the filename in a comment on every reference line (file refs, build files, group children, source phases). Could formalise this as a reusable recipe: "bulk-delete N files from pbxproj."
- **Compiler-driven purge beats grep-driven purge** — 86 refs in AppState but only 7 surfaced as errors because Swift stops early. Iterating on actual errors was measurably faster than preemptive scrubbing. General principle, not project-specific.
- **Two-stage image dedup (filename → pHash)** — filename-based pass on CMS responsive ladders is ~85× more effective than pHash alone (this corpus: 4,368 filename dupes vs 2 visual dupes). Run filename first, it's free; pHash only earns its keep after. Potential cookbook entry: "dedup a photographer archive exported by a CMS."
- **dHash vs pHash failure modes** — dHash collides on low-information images (night shots, flat posters) because its sign-bit gradients go to noise. pHash's DCT-domain signature handles them correctly. For photo archives, skip straight to pHash.
- **Pure-Python DCT is fine at this scale** — 32×32 grayscale, top-left 8×8 coefficients only, 529 images in 3.8s on CPython. No scipy/numpy dependency needed; ~34M float-multiplies total is trivially cheap.

## Learnings / cookbook candidates (addendum)

- **SwiftUI `HStack` intrinsic-size propagation** — `LazyVGrid(.adaptive(minimum:, maximum:))` publishes its `maximum:` as a preferred width upward through the view tree. `HStack` honours the preference unless an explicit `.frame(maxWidth: .infinity)` overrides it. `DropZoneView` had the modifier; `ImageGridView` didn't. Cookbook entry candidate: "flex-fill pattern for content panes in HSplitView / HStack".
- **`.windowResizability` modes are a footgun** — `.contentSize` (snap to content) vs `.contentMinSize` (floor, resize freely above) — names are close, behaviour is opposite. Default to `.contentMinSize` for productivity apps; reserve `.contentSize` for palette/utility windows.
- **SourceKit indexing lag during multi-agent edits** — when 2–3 subagents edit interdependent files in parallel, SwiftLens/SourceKit diagnostics produce a transient flood of "Cannot find type X" errors for types that exist in the same module. `xcodebuild` always settles to the correct final state. Rule: treat SourceKit as a hint, `xcodebuild` as truth during multi-file refactors.
- **Orphan-file false positive** — when deleting a large inherited file, grep for *every nested type*, not just the top-level struct name. `ExportSettingsView.swift` was "orphaned" at the top level but housed a live `FileSizeEstimateView`. Cookbook candidate: "pre-delete checklist for large inherited files".
- **`ToolbarItem(.principal)` is atomic** — if the content doesn't fit, SwiftUI hides the whole item, not just the overflowing children. For dynamic chip rows in a toolbar, cap visible count and implement your own overflow Menu. Don't rely on SwiftUI to "just break up" an HStack inside a single toolbar item.
- **Agent-parallelism sweet spot** — 2–3 parallel subagents with disjoint file ownership consistently produced clean diffs. The one integration-heavy wave (5B) worked better as a single sequential agent because it touched the seam between 5A's three atoms. Heuristic: parallel when file ownership is strict, sequential when the change crosses module boundaries.
