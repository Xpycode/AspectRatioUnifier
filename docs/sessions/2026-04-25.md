**Two real bugs surfaced** (deferred — see "Open" below):

#### Bug 1 — `RatioFilterChips` cap is hardcoded at 6
- `ContentView.swift:154` — `private let maxVisibleChips = 6`
- The "+23 ⌄" overflow menu **toggles filters**, it doesn't promote chips into the strip. The actives-first sort *does* work, but the strip never grows past 6.
- User mistook the +23 menu for a "show more chips" affordance. With 29 buckets in this archive, 6 visible is too few.
- Three fix paths still open: (a) lift the constant, (b) make adaptive to window width via `ViewThatFits` (macOS-native), (c) re-label the menu so it doesn't read as an expand-chips control.

#### Bug 2 — bucketing fragments naturally-continuous clusters
- `RatioAnalyzer.swift:101-111` — greedy bucketer anchors on the **first** (lowest-ratio) probe in each bucket and extends only while new probes are within ±tol of that anchor.
- Result on this archive: the "3:2" cluster spans 1.45–1.51 (4% wide due to scanner noise) and gets fragmented into **four buckets totaling 413 images** (262 + 79 + 44 + 28) instead of one. No single anchor can cover the cluster's natural width when the cluster is wider than the tolerance window.
- Compounding cosmetic bug: `namedPresetSnap = 0.002` is tighter than `bucketTolerance = 0.01`, so adjacent fragmented buckets show up as "3:2" + "1.49:1" + "1.47:1" + "1.46:1" — visually different despite all being "basically 3:2" by the bucketing tolerance.
- Three fix options laid out: F1 snap-to-named-presets-first, F2 drift-the-anchor (chain-merge with max-span cap), F3 combine both. Recommendation: F2 with 3% max-span — smallest change, biggest win.

### Q1 + Q2 + namedPresetSnap shipped (later in same session)

User picked: Q1 = MRU rotation with rightmost-replacement, Q2 = drift-the-anchor with max-span guard. Web search validated the approach has a textbook name (constrained agglomerative clustering: single-link chain bounded by complete-link diameter cap).

**Q2 — `RatioAnalyzer.swift:96–116` rewritten.** Bucket extends iff (a) `gapToLast ≤ bucketTolerance` (single-link chain) AND (b) `proposedSpan ≤ bucketMaxSpan` (complete-link diameter cap). Both guards are needed: chain-only over-merges, span-only fragments. New constant `Config.Ratio.bucketMaxSpan = 0.05` (5%, sized to comfortably cover the ~4% scanner-noise width of a 35mm-scan 3:2 cluster).

Verified on the 530-image archive: 27 sim-buckets → **18 actual buckets**. The 3:2 cluster consolidated from 4 buckets (28+79+44+262 = 413) into **one bucket of 407 images** at span 4.96% (right at the cap). Bucket 11 (4:3-tail at 1.39–1.46) hits 4.94% and the 5% cap correctly prevented it from chaining into bucket 12.

**Q1 — `AppState.swift` + `ContentView.swift`.** Added `var chipOrder: [UUID] = []` to AppState plus `promoteToChips(_:max:)` (replaces rightmost slot, no-op if already chip). `RatioFilterChips`: cap 6→10, dropped actives-first reordering, `visibleBuckets`/`overflowBuckets` compute off `chipOrder` instead of `buckets`. Menu Button now calls `promoteToChips` then `toggleFilter`. Re-analysis resets chipOrder via `Array(buckets.prefix(10).map(\.id))` since bucket UUIDs regenerate.

**Cosmetic-snap widening (immediate follow-up).** Q2's consolidation pulled the 407-image bucket's mean to 1.4897 — 0.69% from true 3:2, outside the old 0.2% snap. So it labeled "1.49:1 (custom)" instead of "3:2". Widened `Config.Ratio.namedPresetSnap` from `0.002` → `0.025` (2.5% — sized to roughly half the bucketMaxSpan so a bucket's center can drift within its diameter and still snap). Result: chips now read "3:2 407× · 1.43:1 72× · 4:3 15× · 2:3 14× · 0.69:1 3× · 1.55:1 3× · 0.71:1 2× · 3:4 2× · 16:10 2×" — five of nine visible chips snap to named presets. The 1.43:1 cluster correctly *didn't* snap (4.7% from 3:2, outside 2.5%).

The named-preset list is sparse enough that 2.5% never collides — gap from 4:3 (1.333) to 3:2 (1.5) is 12.5%, etc.
